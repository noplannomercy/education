### 3막 개요: 왜 스펙주도개발인가 (20분)

**1막/2막과 뭐가 다른가 (판서)**
```
1막: "Todo 앱 만들어줘"    → AI가 결정
2막: "계획하고 만들어줘"   → AI가 계획+실행
3막: "이 스펙대로 만들어"  → 내가 설계, AI가 실행
```

**AI가 일관성을 잃는 이유**
```
컨텍스트 창 = AI의 작업 기억
  → 500줄 넘어가면 초반 결정 까먹기 시작
  → 기능 추가할수록 기존 코드와 충돌
  → 프로젝트 클수록 AI 마음대로 결정이 늘어남

해결책:
  → 레퍼런스 문서 = AI의 장기 기억
  → 스펙이 코드보다 먼저 = AI가 실수할 공간이 없음
  → 서브에이전트 = 매번 클린 컨텍스트로 시작
```

**커맨드 체인 전체 보여주기**
```
/1.prd          → PRD 작성 (요구사항 확정)
/2.brainstorm   → 아키텍처 설계 대화
/3.1.docs       → 레퍼런스 문서 생성 (DB·API·아키텍처·컨벤션)
/3.5.review-docs → 문서 검토
/4.write-plan   → 마이크로 태스크 분해
/5.1.review-implementation → 계획 검토
/5.5.setup-tasks → TodoWrite 설정
/6.execute-plan  → 실행 (TDD + E2E 자동화)
/7.code-review   → 최종 코드 리뷰
```

> **핵심 한 줄:**
> "AI를 코더로 쓰는 게 아니라 AI를 팀원으로 쓰는 방법입니다."
